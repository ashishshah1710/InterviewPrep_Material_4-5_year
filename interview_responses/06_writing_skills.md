## Writing Clear Technical Documentation

### Best Practices

- **Know your audience:** Tailor detail and language for beginners, peers, or advanced users.
- **Use structured formats:** Leverage headings, lists, tables, and code snippets.
- **Be explicit:** Prefer specific instructions to vague references.
- **Employ active voice:** E.g., *芒聙聹Run the installer,芒聙聺* not *芒聙聹The installer should be run.芒聙聺*
- **Stick to present tense:** It feels more immediate and readable.
- **Avoid unnecessary jargon:** Use simple English unless technical terms are necessary.

#### Documentation Template

```markdown
# [Title]: Purpose/Feature Name

## Overview
Brief description of what and why.

## Usage
Step-by-step instructions.

## Parameters
| Parameter | Description | Example |
|-----------|-------------|---------|
|  timeout  |   Wait time |    10   |

## Examples
Provide code or screenshots.

## Troubleshooting
Common issues and fixes.

## References
Links to related docs or resources.
```

## Writing Effective Jira Tickets and Bug Reports

### Best Practices

- **Summarize clearly:** One concise, specific sentence.
- **Provide steps to reproduce:** Numbered, detailed.
- **Describe expected vs actual behavior.**
- **Attach logs, screenshots, environment info.**
- **Assign appropriate priority and labels.**

#### Template

```
*Summary:*  
[Short description]

*Steps to reproduce:*
1. [First step]
2. [Second step]
3. [etc.]

*Expected result:*  
[What should happen]

*Actual result:*  
[What actually happens]

*Environment:*  
- OS:
- Browser:
- App Version:

*Attachments:*  
[Links, screenshots, logs]
```

---

### Before & After Example

**Before (Weak):**

> App crashes sometimes.

**After (Strong):**

> **Summary:** Crash occurs on login when password field is left blank.
> 
> **Steps to reproduce:**
> 1. Launch the app (v2.1.3) on Windows 10.
> 2. Leave the password field empty.
> 3. Click "Login."
> 
> **Expected behavior:**  
> The app should prompt the user to enter a password.
> 
> **Actual behavior:**  
> The app closes unexpectedly.
> 
> **Attachments:**  
> - Crash log (attached)
> - Screenshot of error message

## Concise Writing Tips

- **Cut unnecessary words:**  
  - Weak: "It is often the case that..."  
  - Strong: "Often..."
- **Prefer strong verbs:**  
  - Weak: "Make changes"  
  - Strong: "Modify"
- **Avoid filler and redundancy:**  
  - Weak: "In order to"  
  - Strong: "To"
- **Don芒聙聶t repeat yourself:**

## Structuring Paragraphs

- **Start with a clear topic sentence:** State the main idea.
- **Support:** Offer facts, reasons, or examples.
- **Conclude/transition:** Wrap up or link to next point.

### Example

**Weak:**

> There are lots of ways to optimize code. You can cache stuff. Also, you can break functions up. Some teams use tools to do this.

**Improved:**

> There are several ways to optimize code for better performance. First, caching frequently used data reduces load times. Second, breaking large functions into smaller, reusable units simplifies maintenance. Additionally, teams can leverage code analysis tools to identify bottlenecks.
